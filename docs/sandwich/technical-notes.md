## Architecture Notes
### Vertical Slice Architecture
Each feature is a self-contained module under `features/`. Complex features (products, orders, payment) have full domain/application/presentation layering. Simple features (banners, categories) use presentation-only with direct DB queries. Start simple, add layers only when needed.

### Clean Architecture Layers
Domain layer contains pure TypeScript with zero dependencies. Application layer has direct Drizzle queries (no repository pattern for CRUD). Presentation layer contains UI components and route handlers. No infrastructure abstraction layer.

### Query/Command Split
Commands (mutations) use Server Actions for form submissions. Store queries use direct query import in Server Components (no HTTP overhead). Admin queries use Route Handlers + TanStack Query (HTTP cached, parallelizable).

### Storefront ISR vs Admin CSR
Storefront uses ISR with Server Components for SEO and static HTML generation. Admin uses CSR with TanStack Query for real-time updates. Route handlers serve admin data via HTTP for caching and parallelization.

### Money as Cents
All prices stored as integer cents in database to avoid floating-point issues. Converted to dollars at API boundaries. Display uses formatCurrency helper. DB: priceCents (integer) → API: priceDollars (number) → Display: formatCurrency(cents).

### Payment Gateway Abstraction
PaymentProvider interface allows swapping Midtrans for other providers (Stripe, Xendit) without touching order logic. Webhook handler verifies signature and updates order status atomically.

### Atomic Order Creation
Order + orderItems + payment record inserted in single database transaction. If any fails, all roll back. Idempotency key prevents duplicate orders from double form-submits.

### SEO Implementation
Structured data using JSON-LD for products (schema.org Product, Offer). Dynamic sitemap generated via Next.js sitemap.ts. Per-page meta tags via generateMetadata. Open Graph tags for social sharing. ISR ensures fresh SEO data without rebuilds.

### Image Management
Product images stored in Vercel Blob. Admin uploads via @vercel/blob client. Images served via Vercel's CDN with automatic optimization. No separate CDN service needed.

## Risks
- **[high]** Midtrans sandbox vs production configuration must be clearly separated to avoid payment issues
- **[medium]** ISR revalidation must be triggered correctly after product/order mutations to avoid stale storefront data
- **[low]** Cart stored in localStorage only - clearing browser data loses cart contents
- **[medium]** No inventory tracking was explicitly dropped - may need to be added later if client requirements change
- **[medium]** Overseas shipping calculation adds complexity - need to define supported regions and carrier integrations
