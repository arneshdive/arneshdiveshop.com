## Architecture Notes
### Vertical Slice Architecture
SPEC.md prescribes vertical slice architecture where each feature is self-contained. Frontend already follows this pattern. Backend should maintain same structure with domain/application/presentation layers only when complexity warrants it. Simple CRUD features can skip domain layer.

### No API routes exist currently
All backend work is greenfield. The app/(store) and app/admin directories have complete UI but no app/api directory. Need to create all API routes from scratch. Consider using Next.js server actions for mutations (per SPEC.md) and route handlers for queries.

### Database schema required
No lib/db directory exists. SPEC.md documents schema but file not created. Tables needed: users, customers, addresses, categories, brands, products, product_variants, orders, order_items, payments, promotions, banners, verification_tokens. Drizzle migrations need to be generated and run against Neon.

### ISR for Storefront, CSR for Admin
Spec calls for Incremental Static Regeneration for storefront pages (products, categories) for performance. Admin dashboard uses client-side rendering with TanStack Query for real-time data. This split architecture is appropriate for the use case.

### Session management approach
jose library is in dependencies suggesting JWT-based sessions. Store session token in HTTP-only cookie. Session should contain user ID and role for RBAC. Neon serverless is stateless so JWT is appropriate over database sessions.

### Guest checkout with auto-registration
Users can checkout without prior registration. During checkout, contact data (name, email, phone) is collected. Upon successful order completion, a customer account is automatically created using this data. The new customer receives login credentials via email. This reduces friction while still building a customer database.

### Simple availability model (no stock quantities)
Products have an availability flag (available/unavailable) instead of stock quantity tracking. This simplifies the system - no stock deductions, no low-stock alerts, no stock returns on cancellation. Admin manually toggles availability. Suitable for a dive shop where inventory management is handled externally or not needed initially.

### No reviews/ratings in initial scope
Product reviews and ratings are deferred to a future phase. Simplifies product detail pages and removes need for reviews table, moderation workflow, and rating aggregation logic.

## Risks
- **[medium]** Midtrans webhook in sandbox may not match production behavior perfectly - test thoroughly
- **[low]** ISR revalidation timing may cause stale product data immediately after admin updates
- **[medium]** Email deliverability to Indonesian ISPs via Mailgun may have issues - verify Mailgun has good regional delivery rates
- **[medium]** Zustand cart state needs careful sync with server-side cart to avoid client-server mismatch
- **[low]** Drizzle ORM with Neon serverless may have connection pooling issues under load - verify connection management
- **[medium]** Auto-registration may create duplicate accounts if guest uses existing email - need email lookup and account linking logic
