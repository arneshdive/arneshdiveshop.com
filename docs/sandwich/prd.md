## Modules & Features
### Authentication
> Status: `partial`

User registration, login, session management, password reset, email verification. Supports both explicit registration and auto-registration during guest checkout.

**Features:**
- `[stated]` Email/password registration with validation
- `[stated]` Secure login with session management (HTTP-only cookies)
- `[stated]` Forgot password flow with email OTP
- `[stated]` Email verification for new accounts
- `[stated]` Role-based access control (customer, admin, super_admin)
- `[stated]` Rate limiting on auth endpoints
- `[stated]` Auto-registration from guest checkout with collected user data

### Product Management
> Status: `partial`

Product catalog with variants, categories, brands. Frontend UI exists for both storefront (product listings, detail pages) and admin (product CRUD). Mock data in lib/data/mock-products.ts. No API routes or database. Uses Vercel Blob for image storage. Simple availability flag instead of stock quantities.

**Features:**
- `[stated]` Product CRUD with image upload to Vercel Blob
- `[stated]` Product variants (size, color) with individual pricing
- `[stated]` Category management with hierarchical structure
- `[stated]` Brand management
- `[stated]` Product search and filtering
- `[stated]` Featured products selection
- `[stated]` Product status (active/inactive)
- `[stated]` Availability flag for products (available/unavailable)

### Shopping Cart
> Status: `partial`

Cart system for logged-in and guest users. Frontend UI exists in app/(store)/cart/. Zustand store in lib/store/cart.ts for client state. Needs server-side persistence and API routes.

**Features:**
- `[stated]` Server-side cart storage (database) for logged-in users
- `[stated]` Cart persistence across sessions
- `[stated]` Add/remove/update cart items
- `[stated]` Cart item quantity validation
- `[stated]` Real-time cart total calculation

### Checkout
> Status: `partial`

Multi-step checkout with guest support and auto-registration. Frontend UI exists in app/(store)/checkout/. Guest users provide contact/shipping data which is used to auto-create an account on order completion.

**Features:**
- `[stated]` Multi-step checkout (Contact → Shipping → Payment → Confirm)
- `[stated]` Shipping address form with Indonesia-specific fields (province, city, postal code)
- `[stated]` Guest checkout without prior registration
- `[stated]` Auto-registration on order completion using collected user data
- `[stated]` Order summary review before payment
- `[stated]` Order confirmation page with details

### Payment

Midtrans payment integration for Indonesian market. No implementation exists yet.

**Features:**
- `[stated]` Midtrans Snap integration for checkout
- `[stated]` Support for credit/debit cards
- `[stated]` Bank transfer (Virtual Account)
- `[stated]` E-wallets (GoPay, OVO, ShopeePay)
- `[stated]` QRIS support
- `[stated]` Webhook handler for payment notifications
- `[stated]` Payment status tracking (pending, paid, failed, expired)

- `[stated]` Order creation from cart
- `[stated]` Order list with filtering (status, date range, search)
- `[stated]` Order detail view with customer info, items, payment
- `[stated]` Order status updates (processing, shipped, delivered, cancelled)
- `[stated]` Audit trail for order status changes
- `[stated]` Export orders to CSV/Excel

## Integrations
- `[stated]` Midtrans Payment Gateway
- `[stated]` Neon PostgreSQL (serverless)
- `[stated]` Mailgun for transactional emails
- `[stated]` Vercel Blob for product images

## Constraints
- `[stated]` Frontend UI is complete and cannot be changed significantly
- `[stated]` Must use existing tech stack (Next.js 16, Drizzle ORM, PostgreSQL, Midtrans)
- `[stated]` Indonesia-specific: Midtrans payment, Indonesian address format (province, city, postal code)
- `[stated]` All prices stored as integer cents to avoid floating-point issues
- `[stated]` Order numbers in format: ORD-YYYYMMDD-XXXX
- `[stated]` Soft deletes for products/customers (not hard deletes)
- `[stated]` Page load time < 3 seconds
- `[stated]` API response time < 500ms
- `[stated]` ISR (Incremental Static Regeneration) for product pages
- `[stated]` Budget: IDR 50-100M (approx. USD 3,000-6,000)
- `[stated]` No stock/inventory quantity tracking - simple availability flag only
- `[stated]` No product reviews/ratings for initial delivery
